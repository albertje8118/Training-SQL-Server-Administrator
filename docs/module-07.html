Module 07: Database Maintenance

Lab 07 Index Rebuild · Statistics · DBCC CHECKDB NovaMart Health
Learning Objectives
  • Explain why index fragmentation degrades query performance
  • Decide when to REBUILD versus REORGANIZE based on fragmentation level
  • Update column statistics so the query optimizer has accurate data
  • Run DBCC CHECKDB to detect and report database consistency errors
  • Build a weekly maintenance routine for the NovaMart database

7.1 Index Fragmentation

Fragmentation occurs naturally as INSERT, UPDATE, and DELETE operations scatter data across index pages. Over time, SQL Server must read more pages to satisfy the same query, increasing I/O and reducing throughput.

Fragmentation LevelRecommended Action
< 5%Skip — no action needed
5% – 30%ALTER INDEX … REORGANIZE
> 30%ALTER INDEX … REBUILD
T-SQL — Detect Fragmentation
SELECT
  OBJECT_NAME(ips.object_id)          AS TableName,
  i.name                              AS IndexName,
  ips.avg_fragmentation_in_percent,
  ips.page_count
FROM sys.dm_db_index_physical_stats
     (DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
JOIN sys.indexes i
  ON ips.object_id = i.object_id
 AND ips.index_id  = i.index_id
WHERE ips.avg_fragmentation_in_percent > 5
  AND ips.page_count > 100
ORDER BY ips.avg_fragmentation_in_percent DESC;
ℹ
page_count filter

Indexes with fewer than 100 pages (800 KB) rarely benefit from maintenance — the I/O savings are negligible. Always filter by page_count > 100 to avoid wasting maintenance window time on tiny indexes.

7.2 REBUILD vs REORGANIZE

REORGANIZEREBUILD
Fragmentation range5% – 30%> 30%
How it worksDefragments leaf-level pages in-place, compacts the existing structureDrops and recreates the index from scratch
LockingOnline — no table lock (all editions)Offline by default (table locked); Enterprise supports ONLINE = ON
StatisticsNOT updated automaticallyUpdated automatically (equivalent to FULLSCAN)
SpeedSlower for heavy fragmentationFastest, most thorough result
T-SQL — REORGANIZE (5–30%)
-- Reorganize all indexes on a table
ALTER INDEX ALL ON dbo.Employee REORGANIZE;
GO

-- Reorganize a specific index
ALTER INDEX IX_Employee_DeptID ON dbo.Employee REORGANIZE;
GO
T-SQL — REBUILD (>30%)
-- Rebuild all indexes on a table (offline — table is locked)
ALTER INDEX ALL ON dbo.Employee REBUILD WITH (ONLINE = OFF);
GO

-- Rebuild a specific index (Enterprise: ONLINE = ON avoids locking)
ALTER INDEX PK_Employee ON dbo.Employee REBUILD WITH (ONLINE = OFF);
GO
⚠
Production Impact

REBUILD with ONLINE = OFF takes a Schema Modification (Sch-M) lock on the table, blocking all reads and writes for the duration. Schedule rebuilds during low-traffic windows. If you are on Standard edition and need zero downtime, use REORGANIZE instead even for high-fragmentation indexes.

7.3 Update Statistics

Statistics are histograms that tell the query optimizer how data is distributed across columns. Stale statistics lead to bad execution plans — often the root cause of "suddenly slow" queries after a data load.

T-SQL — Update Statistics
-- Update statistics for one table (full scan — most accurate)
UPDATE STATISTICS dbo.Employee WITH FULLSCAN;
GO

-- Update all statistics in the current database
EXEC sp_updatestats;
GO

-- Check when statistics were last updated
SELECT
  s.name,
  STATS_DATE(s.object_id, s.stats_id) AS LastUpdated
FROM sys.stats s
WHERE OBJECT_NAME(s.object_id) = 'Employee'
ORDER BY LastUpdated DESC;
MethodCoverageNotes
WITH FULLSCAN100% of rows scannedMost accurate — recommended for maintenance jobs
WITH SAMPLE n PERCENTn% of rowsFaster for very large tables; slightly less accurate
sp_updatestatsAll tables in DBOnly updates stats that have changed since last update — quick pass
Auto-update statisticsAutomaticTriggers when ~20% of rows change — usually sufficient between maintenance runs

7.4 DBCC CHECKDB

DBCC CHECKDB validates the physical and logical integrity of every object in the database. Run it regularly — it is your early warning system for storage failures and corruption.

T-SQL — DBCC CHECKDB
-- Full integrity check — checks allocation, structure, and rows
DBCC CHECKDB (NovaMart) WITH NO_INFOMSGS;
GO

-- Fast check — structure and allocation only (no row-level)
DBCC CHECKDB (NovaMart) WITH PHYSICAL_ONLY, NO_INFOMSGS;
GO

-- Check a single table
DBCC CHECKTABLE ('dbo.Employee');
GO
ℹ
What to look for

A clean database returns: "CHECKDB found 0 allocation errors and 0 consistency errors in database 'NovaMart'." Any non-zero error count requires immediate investigation. Do NOT restart the SQL Server service if CHECKDB reports errors — this can make recovery harder.

OptionEffectWhen to Use
NO_INFOMSGSSuppresses informational messages — only shows errorsAlways — keeps output clean
PHYSICAL_ONLYChecks page structure and allocation maps, skips logical row checksFrequent runs (daily/weekly) — much faster
(no options)Full check including row-level logical consistencyWeekly or monthly — the definitive check

Knowledge Check
  1. An index shows 12% fragmentation and 500 pages. Should you REBUILD or REORGANIZE?
  2. After running ALTER INDEX ALL ON dbo.Employee REORGANIZE, are the statistics automatically updated?
  3. What does WITH NO_INFOMSGS do in a DBCC CHECKDB command?
  4. When is the best time to run index REBUILD on a production system?
  5. What does sp_updatestats skip that UPDATE STATISTICS … WITH FULLSCAN does not?
  1. REORGANIZE — 12% falls in the 5–30% range. REORGANIZE is the appropriate choice and keeps the table online.
  2. No. REORGANIZE does NOT update statistics automatically. You must run UPDATE STATISTICS separately after REORGANIZE.
  3. It suppresses informational messages — only errors and warnings are shown. This keeps the output clean and easy to scan for problems.
  4. During a low-traffic maintenance window — typically overnight on weekends. REBUILD with ONLINE = OFF locks the table, blocking reads and writes.
  5. sp_updatestats skips statistics that have not changed since the last update (uses a change-count threshold). WITH FULLSCAN always rescans all rows regardless of changes.

Mission Briefing — Lab 07

Sunday Maintenance Window for NovaMart

It is Sunday morning. Run the full maintenance routine on NovaMart — detect fragmentation, rebuild or reorganise indexes, refresh statistics, and confirm database integrity with DBCC CHECKDB.

Part A — Detect Fragmentation
  1. T-SQL
    USE NovaMart;
GO
SELECT
  OBJECT_NAME(ips.object_id) AS TableName,
  i.name                     AS IndexName,
  ips.avg_fragmentation_in_percent,
  ips.page_count
FROM sys.dm_db_index_physical_stats
     (DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
JOIN sys.indexes i
  ON ips.object_id = i.object_id
 AND ips.index_id  = i.index_id
WHERE ips.avg_fragmentation_in_percent > 5
  AND ips.page_count > 100
ORDER BY ips.avg_fragmentation_in_percent DESC;
    Result set shows fragmentation percentages. On a small training database, fragmentation may be near 0% — this is normal.
Part B — Run Index Maintenance

  1. Rebuild all indexes on each main table:

    T-SQL
    ALTER INDEX ALL ON dbo.Employee   REBUILD WITH (ONLINE = OFF);
ALTER INDEX ALL ON dbo.Department REBUILD WITH (ONLINE = OFF);
GO
    Commands complete with no errors.
Part C — Update Statistics
  1. T-SQL
    -- Full scan for critical tables
UPDATE STATISTICS dbo.Employee WITH FULLSCAN;
UPDATE STATISTICS dbo.Department WITH FULLSCAN;
GO

-- Update remaining tables
EXEC sp_updatestats;
GO
    sp_updatestats reports "Updated statistics for ..." or "Statistics for table ... are already up-to-date."
Part D — DBCC CHECKDB
  1. T-SQL
    DBCC CHECKDB (NovaMart) WITH NO_INFOMSGS;
GO
    CHECKDB found 0 allocation errors and 0 consistency errors in database 'NovaMart'.

Validation Checklist

  • Fragmentation query executed against sys.dm_db_index_physical_stats
  • ALTER INDEX REBUILD completed on Employee and Department tables
  • UPDATE STATISTICS with FULLSCAN completed
  • sp_updatestats executed successfully
  • DBCC CHECKDB reports 0 allocation errors and 0 consistency errors

Troubleshooting

ProblemLikely CauseFix
