ℹ
NVARCHAR vs VARCHAR

Use NVARCHAR for any column that might store non-ASCII characters (e.g., Indonesian, Chinese, Arabic). Use VARCHAR only when you know the data will always be ASCII and storage space is critical.

4.2 Constraints

Constraints enforce data integrity at the database level — they cannot be bypassed by the application layer.

PRIMARY KEY

Uniquely identifies each row. Automatically creates a clustered index. A table can have only one primary key. NULL values are not allowed.

FOREIGN KEY

Enforces referential integrity between two tables. A value in the child column must exist in the parent table's referenced column (or be NULL). Prevents orphaned rows.

UNIQUE

Ensures all values in a column (or combination of columns) are unique across the table. Unlike PRIMARY KEY, multiple UNIQUE constraints can exist and NULLs are permitted (but only one NULL).

NOT NULL

Ensures a column always has a value — NULL is rejected on insert/update.

DEFAULT

Provides an automatic value if the column is omitted from an INSERT statement.

CHECK

Validates that column values meet a Boolean condition. For example, salary must be greater than 0.

4.3 CREATE TABLE

T-SQL — Department table
USE NovaMart;
GO

CREATE TABLE dbo.Department (
    DeptID   INT          NOT NULL IDENTITY(1,1),
    DeptName NVARCHAR(50) NOT NULL,
    Location NVARCHAR(50) NOT NULL,
    CONSTRAINT PK_Department PRIMARY KEY (DeptID),
    CONSTRAINT UQ_Department_Name UNIQUE (DeptName)
);
GO
T-SQL — Employee table with FK and CHECK
CREATE TABLE dbo.Employee (
    EmpID    INT           NOT NULL IDENTITY(1,1),
    FullName NVARCHAR(100) NOT NULL,
    DeptID   INT           NOT NULL,
    Salary   DECIMAL(12,2) NOT NULL,
    HireDate DATE          NOT NULL DEFAULT GETDATE(),
    Email    NVARCHAR(150) NULL,
    CONSTRAINT PK_Employee    PRIMARY KEY (EmpID),
    CONSTRAINT FK_Emp_Dept    FOREIGN KEY (DeptID) REFERENCES dbo.Department (DeptID),
    CONSTRAINT CHK_Emp_Salary CHECK (Salary > 0),
    CONSTRAINT UQ_Emp_Email   UNIQUE (Email)
);
GO
ℹ
IDENTITY(1,1)

IDENTITY(seed, increment) auto-generates sequential integers. IDENTITY(1,1) starts at 1 and increments by 1. You cannot insert into an IDENTITY column directly unless you first run SET IDENTITY_INSERT tablename ON.

4.5 SELECT — Querying Data

T-SQL
-- All columns
SELECT * FROM dbo.Employee;

-- Specific columns with alias
SELECT EmpID, FullName AS Name, Salary
FROM   dbo.Employee;

-- Filter and sort
SELECT EmpID, FullName, Salary
FROM   dbo.Employee
WHERE  Salary > 6000000
ORDER  BY Salary DESC;

-- INNER JOIN: employees with their department names
SELECT e.EmpID, e.FullName, d.DeptName, e.Salary
FROM   dbo.Employee    e
INNER JOIN dbo.Department d ON e.DeptID = d.DeptID
ORDER  BY d.DeptName, e.FullName;

-- Aggregate: average salary by department
SELECT d.DeptName, COUNT(e.EmpID) AS HeadCount,
       AVG(e.Salary) AS AvgSalary
FROM   dbo.Employee    e
INNER JOIN dbo.Department d ON e.DeptID = d.DeptID
GROUP  BY d.DeptName
ORDER  BY d.DeptName;

4.6 UPDATE and DELETE

T-SQL
-- UPDATE: give Sari a raise
UPDATE dbo.Employee
SET    Salary = 7000000
WHERE  FullName = 'Sari Dewi';

-- Verify the change
SELECT FullName, Salary FROM dbo.Employee WHERE FullName = 'Sari Dewi';

-- DELETE: remove a specific row
DELETE FROM dbo.Employee WHERE EmpID = 99;

-- IMPORTANT: always use a WHERE clause with UPDATE and DELETE
-- Running UPDATE or DELETE without WHERE will affect ALL rows
⚠
Always Use a WHERE Clause

An UPDATE or DELETE statement without a WHERE clause affects every row in the table. Before running a destructive DML, use SELECT with the same WHERE clause to confirm which rows will be affected.

4.7 Basic Transactions

Wrapping DML in an explicit transaction allows you to review changes before committing — a safety net for data corrections.

T-SQL
BEGIN TRANSACTION;

UPDATE dbo.Employee
SET    Salary = 8000000
WHERE  DeptID = 1;

-- Check the changes before committing
SELECT * FROM dbo.Employee WHERE DeptID = 1;

-- If the result is correct:
COMMIT TRANSACTION;

-- If the result is wrong:
-- ROLLBACK TRANSACTION;
Knowledge Check
  1. You need to store an employee's full name that may include Indonesian or Chinese characters. Which data type is most appropriate?
  2. What constraint prevents a row in dbo.Employee from referencing a non-existent DeptID?
  3. You run INSERT INTO dbo.Employee (DeptID, Salary) VALUES (1, 0). Will the insert succeed? Why or why not?
  4. How do you select all employees who earn more than 6,000,000 and sort them from highest to lowest salary?
  5. What is the risk of running DELETE FROM dbo.Employee without a WHERE clause?
  1. NVARCHAR(n) — supports Unicode characters
  2. FOREIGN KEY constraint on DeptID referencing dbo.Department(DeptID)
  3. No — the CHK_Emp_Salary CHECK constraint requires Salary > 0; a salary of 0 violates this constraint. Also FullName is NOT NULL so that column would fail too.
  4. SELECT * FROM dbo.Employee WHERE Salary > 6000000 ORDER BY Salary DESC;
  5. It deletes every row in the table — all employee data is lost

Mission Briefing — Lab 04

Building the HR Module for Sari

Sari (HR Manager) needs a place to store employee and department data. Previously this was an Excel spreadsheet on her desktop — no backups, no enforced business rules. Today you create the first production tables inside NovaMart with proper constraints.

After this lab, Sari will be able to query her own data — but first you need to build the structure and load the sample data she provided.

Part D — Test Constraints

  1. Try to insert an employee with an invalid DeptID:

    T-SQL
    -- This should FAIL with a foreign key violation
INSERT INTO dbo.Employee (FullName, DeptID, Salary, HireDate)
VALUES ('Test User', 99, 5000000, '2024-01-01');
GO
    Error: The INSERT statement conflicted with the FOREIGN KEY constraint "FK_Emp_Dept".

  2. Try to insert an employee with a negative salary:

    T-SQL
    -- This should FAIL with a CHECK constraint violation
INSERT INTO dbo.Employee (FullName, DeptID, Salary, HireDate)
VALUES ('Test User', 1, -1000, '2024-01-01');
GO
    Error: The INSERT statement conflicted with the CHECK constraint "CHK_Emp_Salary".
Part E — UPDATE, DELETE, and JOIN Query

  1. Update Sari's salary:

    T-SQL
    UPDATE dbo.Employee
SET    Salary = 7000000
WHERE  FullName = 'Sari Dewi';
GO
    1 row affected.

  2. Run a join query to produce a combined report:

    T-SQL
    SELECT e.EmpID, e.FullName, d.DeptName, d.Location,
       e.Salary, e.HireDate
FROM   dbo.Employee    e
INNER JOIN dbo.Department d ON e.DeptID = d.DeptID
ORDER  BY d.DeptName, e.FullName;
    6 rows with department names and locations joined in. Sari's salary shows 7,000,000.

Validation Checklist

  • dbo.Department table created with PK + UNIQUE constraint
  • 4 departments inserted successfully
  • dbo.Employee table created with PK, FK, CHECK, UNIQUE constraints
  • 6 employees inserted successfully
  • Insert with invalid DeptID = 99 fails with FK error
  • Insert with Salary = -1000 fails with CHECK error
  • JOIN query returns 6 rows with correct department names
