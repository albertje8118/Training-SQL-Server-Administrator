Module 05: Security & User Management

Lab 05 Logins · Roles · GRANT Sari's Access
Learning Objectives
  • Explain the two-level security model: server-level login vs database-level user
  • Create a SQL Server login and map it to a database user
  • Assign fixed server roles and fixed database roles
  • Use GRANT, DENY, and REVOKE to control object-level permissions
  • Test that permissions are enforced correctly

5.1 The Two-Level Security Model

SQL Server enforces a two-level security architecture. Understanding this is critical before creating any user accounts.

LevelObjectScopeControls
ServerLoginInstance-wideWhether the connection is allowed to reach the instance at all
DatabaseUserSingle databaseWhat the login can do inside a specific database
ℹ
Login vs User

A Login authenticates the connection to the SQL Server instance (server-level). A User authorizes what that login can do inside a specific database (database-level). A login can be mapped to users in multiple databases, but typically one login → one user per database.

5.2 Login Types

TypeAuthenticationWhen to Use
Windows Login (User)Windows/KerberosDomain users connecting from domain-joined workstations — preferred
Windows Login (Group)Windows/KerberosEntire AD group granted access (e.g., DOMAIN\SQL_Readers)
SQL Server LoginSQL credentialsApplication accounts, cross-domain scenarios, development/testing

5.3 Fixed Server Roles

Server roles control access at the instance level. They are pre-defined — you cannot create custom server roles in most editions.

RolePermissions
sysadminFull control over the entire instance — unrestricted. Assign only to DBAs.
serveradminConfigure server settings, shut down the instance
securityadminManage logins and login permissions — can effectively grant themselves sysadmin
processadminKill processes (KILL command)
dbcreatorCreate, alter, drop, and restore any database
diskadminManage disk files
bulkadminExecute BULK INSERT statements
publicMinimum permissions — all logins are members by default
⚠
Principle of Least Privilege

Assign only the permissions a user actually needs. Avoid granting sysadmin to application accounts or regular users. Regularly audit login memberships.

5.4 Fixed Database Roles

Database roles control access within a single database. The most commonly used:

RoleEffective Permissions
db_ownerFull control of the database — treat like a DBA for that DB
db_datareaderSELECT on all user tables and views in the database
db_datawriterINSERT, UPDATE, DELETE on all user tables
db_ddladminCreate, alter, drop schema objects (tables, indexes, views)
db_securityadminManage database role membership and permissions
db_backupoperatorBack up the database
db_denydatareaderExplicitly denies SELECT — useful to block access to specific tables while allowing other permissions
db_denydatawriterExplicitly denies INSERT/UPDATE/DELETE
publicMinimum permissions — all database users are members by default

5.5 Creating a Login

T-SQL — SQL Server Login
-- Create a SQL Server login
CREATE LOGIN LabReader
WITH PASSWORD    = 'P@ssw0rd_Lab!',
     CHECK_POLICY = ON,   -- enforce Windows password policy
     CHECK_EXPIRATION = OFF;
GO
T-SQL — Windows Login
-- Create a login for a Windows/AD account
CREATE LOGIN [DOMAIN\sari.dewi] FROM WINDOWS;
GO
ℹ
CHECK_POLICY

Setting CHECK_POLICY = ON enforces Windows password complexity rules on the SQL login. Always set this to ON for production logins — do not create logins with blank or trivial passwords.

5.6 Mapping a Login to a Database User

T-SQL
-- Switch to the target database
USE NovaMart;
GO

-- Create a database user mapped to the login
CREATE USER LabReader FOR LOGIN LabReader;
GO

By default the new user has only public permissions — they cannot read any tables yet.

5.7 Assigning Database Roles

T-SQL
-- Grant read-only access (all tables in the database)
ALTER ROLE db_datareader ADD MEMBER LabReader;
GO

-- Grant read/write access (add db_datawriter as well)
ALTER ROLE db_datawriter ADD MEMBER LabReader;
GO

-- Remove role membership
ALTER ROLE db_datawriter DROP MEMBER LabReader;
GO

5.8 GRANT, DENY, and REVOKE

For more granular control beyond fixed roles, use explicit permission statements.

StatementEffect
GRANTAllows the permission. Can be overridden by DENY.
DENYExplicitly blocks the permission. DENY always wins — even if a role also grants it.
REVOKERemoves a previously granted or denied permission. Returns the principal to whatever permissions exist from role membership.
T-SQL — Object-level permissions
USE NovaMart;
GO

-- Allow LabReader to SELECT from a specific table only
GRANT SELECT ON dbo.Employee TO LabReader;

-- Block LabReader from seeing the Department table (even if db_datareader is granted)
DENY SELECT ON dbo.Department TO LabReader;

-- Remove the DENY to restore default role-based access
REVOKE SELECT ON dbo.Department FROM LabReader;
GO

5.9 Testing Permissions with EXECUTE AS

Use EXECUTE AS LOGIN or EXECUTE AS USER to temporarily assume another identity and test permissions — without logging out of your session.

T-SQL
-- Impersonate LabReader
EXECUTE AS LOGIN = 'LabReader';
GO

-- Test: this should succeed if LabReader has db_datareader
SELECT TOP 5 * FROM dbo.Employee;

-- Test: this should fail (INSERT not allowed for db_datareader)
INSERT INTO dbo.Employee (FullName, DeptID, Salary, HireDate)
VALUES ('Test', 1, 5000000, '2024-01-01');

-- Return to your original login
REVERT;
GO

5.10 Viewing Permissions

T-SQL
-- All server logins
SELECT name, type_desc, is_disabled, create_date
FROM   sys.server_principals
WHERE  type IN ('S','U','G')
ORDER  BY type_desc, name;

-- Role membership for a specific login
SELECT r.name AS RoleName
FROM   sys.server_role_members rm
JOIN   sys.server_principals r ON rm.role_principal_id = r.principal_id
JOIN   sys.server_principals m ON rm.member_principal_id = m.principal_id
WHERE  m.name = 'LabReader';

-- Database-level permissions for a user
USE NovaMart;
SELECT dp.name, p.permission_name, p.state_desc, p.class_desc, p.major_id
FROM   sys.database_permissions p
JOIN   sys.database_principals  dp ON p.grantee_principal_id = dp.principal_id
WHERE  dp.name = 'LabReader';

-- Current user context
SELECT SYSTEM_USER AS Login, USER_NAME() AS DBUser;
SELECT IS_MEMBER('db_datareader') AS IsMemberOfDatareader;

  4. How do you temporarily test another user's permissions without logging out?
  5. Which system view shows all server-level logins?

Mission Briefing — Lab 05

Access Control for Sari, Agus, and Dian

Budi has sent over the access requirements:

  • Sari (HR) — read-only access to NovaMart. Uses a SQL login.
  • Agus (Warehouse) — read and write access to NovaMart. Uses a SQL login.
  • Dian (Finance) — read-only, but must not see salary data. Uses a SQL login.

In a real deployment these would be Windows accounts. For the training lab we use SQL logins so the exercise works on any machine without a domain controller.

Part A — Create Logins
  1. T-SQL
    -- Sari: read-only HR
CREATE LOGIN LabSari
WITH PASSWORD = 'P@ssw0rd_Sari!', CHECK_POLICY = ON, CHECK_EXPIRATION = OFF;

-- Agus: warehouse read/write
CREATE LOGIN LabAgus
WITH PASSWORD = 'P@ssw0rd_Agus!', CHECK_POLICY = ON, CHECK_EXPIRATION = OFF;

-- Dian: finance read-only (restricted salary view)
CREATE LOGIN LabDian
WITH PASSWORD = 'P@ssw0rd_Dian!', CHECK_POLICY = ON, CHECK_EXPIRATION = OFF;
GO
    3 commands, each completing successfully.
Part B — Create Database Users
  1. T-SQL
    USE NovaMart;
GO

CREATE USER LabSari FOR LOGIN LabSari;
CREATE USER LabAgus FOR LOGIN LabAgus;
CREATE USER LabDian FOR LOGIN LabDian;
GO
    3 commands completed successfully. In Object Explorer: NovaMart → Security → Users now shows LabSari, LabAgus, LabDian.
Part C — Assign Roles
  1. T-SQL
    -- Sari: read-only
ALTER ROLE db_datareader ADD MEMBER LabSari;

-- Agus: read and write
ALTER ROLE db_datareader ADD MEMBER LabAgus;
ALTER ROLE db_datawriter ADD MEMBER LabAgus;

-- Dian: read-only
ALTER ROLE db_datareader ADD MEMBER LabDian;
GO
    Commands complete successfully.
Part D — Restrict Dian's Access to the Salary Column

  1. Dian should not see salary figures. Use column-level DENY:

    T-SQL
    -- Deny Dian access to the Salary column specifically
DENY SELECT ON dbo.Employee (Salary) TO LabDian;
GO
Part E — Test Permissions

  1. Test Sari's read access:

    T-SQL
    EXECUTE AS LOGIN = 'LabSari';
GO
SELECT * FROM dbo.Employee;     -- should succeed
SELECT * FROM dbo.Department;   -- should succeed
INSERT INTO dbo.Department (DeptName, Location) VALUES ('Test', 'Office'); -- should FAIL
REVERT;
GO
    SELECT succeeds; INSERT fails with permission denied.

  2. Test Agus's read/write access:

    T-SQL
    EXECUTE AS LOGIN = 'LabAgus';
GO
INSERT INTO dbo.Department (DeptName, Location) VALUES ('TestDept', 'Office'); -- should succeed
DELETE FROM dbo.Department WHERE DeptName = 'TestDept';   -- should succeed
REVERT;
GO
    Both statements succeed for Agus.

  3. Test Dian's column restriction:

    T-SQL
    EXECUTE AS LOGIN = 'LabDian';
GO
SELECT EmpID, FullName, DeptID, HireDate FROM dbo.Employee;  -- should succeed
SELECT EmpID, FullName, Salary            FROM dbo.Employee;  -- should FAIL
REVERT;
GO
    First query succeeds. Second query fails: "The SELECT permission was denied on the column 'Salary'".
Part F — Connect as LabSari in a New SSMS Window (Optional)

  1. Open a new SSMS connection: File → Connect Object Explorer.

  2. Use SQL Server Authentication, enter login LabSari and the password you set. Connect.

    Connected. Note that Object Explorer shows only databases LabSari can see — limited compared to sysadmin.

  3. Run SELECT * FROM dbo.Employee; and INSERT INTO dbo.Department... to confirm the read/write boundary.

Validation Checklist

  • Logins LabSari, LabAgus, LabDian created successfully
  • Database users created and mapped to logins in NovaMart
  • LabSari has db_datareader only
  • LabAgus has both db_datareader and db_datawriter
  • LabSari SELECT succeeds; INSERT fails with "permission denied"
  • LabAgus INSERT and DELETE both succeed
  • LabDian SELECT without Salary column succeeds; SELECT including Salary fails
