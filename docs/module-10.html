10.1 Common Connection Errors

Connection failures are the most frequent first call a DBA receives. Match the error number to a targeted fix rather than guessing.

ErrorMessageLikely CauseFix
53A network-related error while establishing a connectionSQL Server not running, wrong server name, firewall blocking port 1433Check service status in Configuration Manager; verify firewall rules; confirm server name
17SQL Server does not exist or access deniedInstance name incorrect, SQL Browser service stopped (named instances)Verify instance name; start SQL Server Browser service
4060Cannot open database requested by the loginDatabase does not exist, is offline, or user has no accessVerify database name; check database state; grant user access
10060A connection attempt failed because the connected party did not respondTCP/IP protocol disabled, wrong port, or network timeoutEnable TCP/IP in Configuration Manager; verify port; check network path
ℹ
First Checks for Any Connection Failure
  1. Is the SQL Server service running? (Configuration Manager or Services.msc)
  2. Is TCP/IP protocol enabled? (Configuration Manager → SQL Server Network Configuration)
  3. Is port 1433 open in Windows Firewall?
  4. Are you using the correct server\instance name?

10.2 Login Failure — Error 18456

Error 18456 always means login failed, but the state code reveals the specific cause. Look at the state number in the error log — the message shown to the user deliberately hides this detail for security reasons.

StateMeaningFix
1Generic — check error log for real stateOpen SQL Server error log to find the actual state code
2Invalid user ID (login not found)Create the login or fix the typo in the connection string
5Login is invalid (not provisioned on this instance)Create the SQL login: CREATE LOGIN … WITH PASSWORD
6Login attempted to use Windows authentication with a SQL login nameVerify authentication mode; switch to SQL authentication
7Login is disabledALTER LOGIN [name] ENABLE;
8Incorrect passwordReset password: ALTER LOGIN [name] WITH PASSWORD = '…';
9Invalid password (does not meet policy)Set a password that meets complexity requirements
11 / 12Valid login, but does not have server access / denied explicitlyGrant server access or remove the DENY permission
18Password must be changed (MUST_CHANGE is set)Connect with SSMS and change the password, or: ALTER LOGIN [name] WITH PASSWORD = '…' OLD_PASSWORD = '…';
38Database does not exist or user cannot access itVerify the default database exists and the user has access
T-SQL — Enable and Unlock a Login
-- Enable a disabled login
ALTER LOGIN [novauser] ENABLE;
GO

-- Reset password for a SQL login
ALTER LOGIN [novauser] WITH PASSWORD = 'NewP@ssw0rd2026';
GO

-- Unlock an account locked by the password policy
ALTER LOGIN [novauser] WITH PASSWORD = 'NewP@ssw0rd2026' UNLOCK;
GO

-- Set a new default database
ALTER LOGIN [novauser] WITH DEFAULT_DATABASE = [NovaMart];
GO

10.3 Reading the SQL Server Error Log

The SQL Server error log captures service start/stop events, login failures (with state codes), backup completions, integrity check results, and configuration changes. Always read the error log before making changes.

MethodSteps
SSMS GUIObject Explorer → Management → SQL Server Logs → double-click Current
T-SQLEXEC sp_readerrorlog; — returns all entries from the current log file
Filtered T-SQLEXEC sp_readerrorlog 0, 1, 'Login failed'; — filter by keyword
T-SQL — Read and Filter Error Log
-- Read the current error log (log file 0)
EXEC sp_readerrorlog;
GO

-- Filter for login failures
EXEC sp_readerrorlog 0, 1, N'Login failed';
GO

-- Filter for a specific login name
EXEC sp_readerrorlog 0, 1, N'novauser';
GO

-- Cycle (archive) the current log — creates a new empty current log
EXEC sp_cycle_errorlog;
GO
ℹ
How Many Log Files?

SQL Server keeps 6 archived error logs by default (plus the current one). Each SQL Server restart creates a new current log. You can increase the number of archived logs in SSMS: Object Explorer → Management → SQL Server Logs → right-click → Configure → increase the count to 20–30 for production.

10.4 Performance Bottleneck Categories

When a query or workload is slow, narrow the cause to one of these six categories before optimising. Mixing symptoms and fixes from different categories wastes time.

CategorySymptomsFirst Steps
CPUHigh CPU %, SOS_SCHEDULER_YIELD waits, slow compilationFind top CPU queries via dm_exec_query_stats; optimise or parameterise
MemoryPage Life Expectancy dropping, PAGEIOLATCH waits, SQL Server paging to diskCheck PLE in Activity Monitor; add RAM; tune max server memory
I/OLong disk read/write times in Data File I/O pane, PAGEIOLATCH_SH/EXMove files to faster disks; separate data/log files; add indexes to reduce scans
Locks / BlockingSessions stuck in SUSPENDED state, LCK_M_X waits, user reports "spinning"Identify head blocker via dm_exec_requests; optimise transaction length; consider SNAPSHOT isolation
NetworkASYNC_NETWORK_IO waits, result sets take longer to arrive than to runReduce result set size; check client-side processing; check network bandwidth
Query DesignTable scans, implicit conversions, missing indexes, bad cardinality estimatesReview execution plans; add missing indexes; fix implicit conversions; update statistics
Knowledge Check
  1. A developer gets "Error 18456, State 7" when connecting. What is the problem and how do you fix it?
  2. Which SSMS location shows the current SQL Server error log without running T-SQL?
  3. A user gets connection error 53. What are the first two things you check?
  4. What performance category would you investigate first if dm_os_wait_stats shows PAGEIOLATCH_SH as the dominant wait?
  5. What does EXEC sp_cycle_errorlog do?
  1. State 7 means the login is disabled. Fix: ALTER LOGIN [loginname] ENABLE;
  2. Object Explorer → Management → SQL Server Logs → Current — double-click to open the Log File Viewer.
  3. 1) Is the SQL Server service running? (check in Configuration Manager or Services.msc) 2) Is port 1433 open in Windows Firewall?
  4. I/O or Memory — PAGEIOLATCH waits indicate SQL Server is waiting to read pages from disk, usually because the buffer pool is too small or storage is slow.
  5. It archives the current error log and starts a new empty one. This is safe to run on a running instance and does not clear old archived logs.

Resolve Three NovaMart Support Tickets
~40 min Requires: NovaMart DB · SSMS · sysadmin access Error Log · Logins · Query
Mission Briefing — Lab 10

Monday Morning Support Queue

Three support tickets landed in the queue this morning. Work through each one: diagnose the root cause using the tools from this module, apply the fix, and verify the resolution.

Ticket 1 — "Cannot connect to SQL Server from the app server"

  1. Check that the SQL Server service is running:

    PowerShell
    Get-Service -Name 'MSSQLSERVER','SQLSERVERAGENT' | Select-Object Name,Status
    Both services show Status = Running.

  2. Open SQL Server Configuration Manager. Confirm TCP/IP is enabled under SQL Server Network Configuration → Protocols for MSSQLSERVER.

    TCP/IP shows Enabled.

  3. Verify the Windows Firewall rule for port 1433:

    PowerShell
    Get-NetFirewallRule -DisplayName '*SQL*' | Select-Object DisplayName, Enabled, Action
    A rule for SQL Server (TCP 1433) is present and Enabled.
Ticket 2 — "Login 'novauser' fails with Error 18456"

  1. First, read the error log to find the state code:

    T-SQL
    EXEC sp_readerrorlog 0, 1, N'novauser';
    One or more rows appear. Read the message column — look for "State: 7" (login disabled) or another state code.

  2. Simulate the issue — disable the login, then fix it:

    T-SQL
    -- Simulate: disable novauser (if it exists)
-- ALTER LOGIN [novauser] DISABLE;

-- Fix: enable the login
ALTER LOGIN [novauser] ENABLE;
GO

-- Confirm it is enabled
SELECT name, is_disabled FROM sys.server_principals WHERE name = 'novauser';
    is_disabled = 0 (login is enabled).
Ticket 3 — "Query on Employee table is very slow"

  1. Enable Actual Execution Plan (Ctrl+M) then run the slow query:

    T-SQL
    USE NovaMart;
GO
SET STATISTICS IO ON;
SET STATISTICS TIME ON;
GO

SELECT * FROM dbo.Employee WHERE Salary > 5000000;
GO

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
GO
    Check the Execution Plan tab — note if a Table Scan is present. Check Messages tab for logical reads count.

  2. Review wait statistics to confirm no blocking:

    T-SQL
    SELECT r.session_id, r.wait_type, r.cpu_time,
       SUBSTRING(t.text,1,200) AS QueryText
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.session_id > 50
ORDER BY r.cpu_time DESC;
    Your session's wait_type is NULL or RESOURCE_SEMAPHORE — no lock blocking. The slow query is a query design issue (Table Scan).

Validation Checklist

  • SQL Server and SQL Agent services confirmed running via PowerShell
  • TCP/IP protocol confirmed Enabled in Configuration Manager
  • sp_readerrorlog used to find login failure state code
  • novauser login enabled and is_disabled = 0 confirmed
  • Execution plan opened and Table Scan operator identified
  • dm_exec_requests consulted to rule out blocking as cause
