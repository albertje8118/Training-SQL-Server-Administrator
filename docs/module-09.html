9.1 Activity Monitor

Activity Monitor is the quickest way to get a snapshot of server health in SSMS. Open it with Ctrl+Alt+A or right-click the server in Object Explorer → Activity Monitor.

PaneWhat It Shows
OverviewCPU, Wait, I/O, Batch Requests/sec — the at-a-glance health dials
ProcessesAll active sessions — SPID, login, database, status, wait type, CPU, memory
Resource WaitsCumulative wait time grouped by category — spot the dominant bottleneck
Data File I/OReads, writes, and response time per database file — detect slow disks
Recent Expensive QueriesTop queries by CPU/duration from the plan cache
ℹ
Activity Monitor Overhead

Activity Monitor queries system views every 10 seconds by default. On a busy production server this is negligible, but set the refresh interval to 30–60 seconds if you are concerned. You can also pause the refresh while reading a snapshot.

9.2 Key Dynamic Management Views

DMVs provide real-time server diagnostics through standard T-SQL queries. They are the foundation of every DBA monitoring script.

T-SQL — Active Requests (dm_exec_requests)
-- Show all currently executing sessions with their query text
SELECT
  r.session_id,
  r.status,
  r.wait_type,
  r.wait_time / 1000.0         AS WaitSec,
  r.cpu_time                   AS CPU_ms,
  r.total_elapsed_time / 1000  AS ElapsedSec,
  r.blocking_session_id,
  SUBSTRING(t.text, 1, 300)    AS QueryText
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.session_id > 50        -- skip system sessions
ORDER BY r.total_elapsed_time DESC;
T-SQL — Top Wait Statistics (dm_os_wait_stats)
-- Top 10 waits by total wait time since SQL Server last started
SELECT TOP 10
  wait_type,
  waiting_tasks_count,
  CAST(wait_time_ms / 1000.0 AS DECIMAL(10,1)) AS TotalWaitSec,
  CAST(signal_wait_time_ms * 100.0 / wait_time_ms AS DECIMAL(5,1)) AS SignalPct
FROM sys.dm_os_wait_stats
WHERE wait_type NOT IN (
  'SLEEP_TASK','BROKER_TO_FLUSH','BROKER_EVENTHANDLER',
  'REQUEST_FOR_DEADLOCK_SEARCH','HADR_WORK_QUEUE','CLR_AUTO_EVENT',
  'DISPATCHER_QUEUE_SEMAPHORE','XE_DISPATCHER_WAIT','FT_IFTS_SCHEDULER_IDLE_WAIT',
  'SQLTRACE_BUFFER_FLUSH','CHECKPOINT_QUEUE','DBMIRROR_EVENTS_QUEUE',
  'SQLTRACE_INCREMENTAL_FLUSH_SLEEP','WAITFOR','SLEEP_DBSTARTUP',
  'SLEEP_MASTERDBREADY','SLEEP_MASTERMDREADY','SLEEP_MASTERUPGRADED',
  'SLEEP_MSDBSTARTUP','SLEEP_SYSTEMTASK','SLEEP_TEMPDBSTARTUP',
  'SNI_HTTP_ACCEPT','SP_SERVER_DIAGNOSTICS_SLEEP','SQLTRACE_WAIT_ENTRIES',
  'WAIT_XTP_OFFLINE_CKPT_NEW_LOG','XE_TIMER_EVENT'
)
ORDER BY wait_time_ms DESC;
T-SQL — Buffer Pool Usage by Database
-- How much memory each database is using in the buffer pool
SELECT
  DB_NAME(database_id) AS DatabaseName,
  COUNT(*) * 8 / 1024  AS BufferMB
FROM sys.dm_os_buffer_descriptors
GROUP BY database_id
ORDER BY BufferMB DESC;
DMVPurpose
sys.dm_exec_requestsCurrently executing requests — CPU, wait type, blocking
sys.dm_exec_sessionsAll connected sessions (active and idle)
sys.dm_os_wait_statsCumulative wait statistics since last restart — identifies dominant bottleneck
sys.dm_os_buffer_descriptorsBuffer pool contents — memory usage per database
sys.dm_exec_query_statsAggregated performance stats per cached query plan
sys.dm_exec_sql_textReturns the T-SQL text for a given sql_handle (used with CROSS APPLY)

9.3 Blocking and Deadlocks

Blocking occurs when one session holds a lock that another session needs. The waiting session is blocked until the first session commits or rolls back. In a blocking chain, there is always one head blocker — the session that started the problem.

T-SQL — Identify Blocking Chain
-- Sessions that are blocked and what is blocking them
SELECT
  r.session_id                 AS BlockedSPID,
  r.blocking_session_id        AS BlockedBy,
  r.wait_type,
  r.wait_time / 1000.0         AS WaitSec,
  SUBSTRING(t.text, 1, 200)    AS BlockedQuery,
  s.login_name,
  s.host_name
FROM sys.dm_exec_requests r
JOIN sys.dm_exec_sessions   s ON r.session_id = s.session_id
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.blocking_session_id <> 0
ORDER BY r.wait_time DESC;

-- Kill a blocking session (use with caution)
-- KILL 52;  -- replace 52 with the head blocker SPID
⚠
KILL with Caution

Running KILL <SPID> terminates the session and rolls back its open transaction. This can take time for large transactions. Always identify why blocking is happening (long-running transaction, missing indexes, bad query plan) rather than repeatedly killing sessions.

9.4 Common Wait Statistics

Wait TypeIndicatesCommon Fix
PAGEIOLATCH_SH / _EXWaiting to read a page from disk — I/O pressure or insufficient memoryAdd RAM, move to faster disk, add indexes to reduce I/O
LCK_M_XWaiting for an exclusive lock — blocking or deadlocksOptimise transactions, add row-level locking, use SNAPSHOT isolation
CXPACKETParallel query threads waiting for each other — skewed parallelismTune MAXDOP setting, add statistics, improve query or indexes
ASYNC_NETWORK_IOSQL Server is waiting for the client to consume results — slow network or large result setsReturn fewer rows, paginate results, check network bandwidth
SOS_SCHEDULER_YIELDThreads voluntarily yielding CPU — heavy CPU pressureIdentify top CPU queries, optimise or cache results
WRITELOGWaiting for transaction log to be flushed to disk — slow log diskMove log files to dedicated fast disk (SSD recommended)

9.5 Reading Execution Plans

An execution plan shows the physical operations SQL Server uses to execute a query — table scans, index seeks, joins. Read plans right to left — data flows from leaf operators (data access) to the root (final output).

Operator / WarningMeaningAction
Table ScanReads every row in the table — no useful index availableAdd an index on the filtered column(s)
Index ScanReads entire index — returning many rows or no SARGable predicateReview WHERE clause; consider a covering index
Index SeekNavigates the B-tree directly — efficient access ✓No action needed — this is good
Key LookupNon-clustered index seek + extra round-trip to clustered index for additional columnsAdd included columns to the non-clustered index
Yellow warning triangleMissing statistics, implicit conversion, or missing index hintHover to read the warning text and act accordingly
T-SQL — View Actual Execution Plan
-- Option 1: Turn on actual execution plan in SSMS toolbar (Ctrl+M)
-- Then run your query — the plan appears in the Execution Plan tab

-- Option 2: Use SET STATISTICS to view time and I/O cost
SET STATISTICS TIME ON;
SET STATISTICS IO ON;
GO

SELECT e.FullName, d.DeptName
FROM   dbo.Employee   e
JOIN   dbo.Department d ON e.DeptID = d.DeptID
WHERE  e.Salary > 5000000;
GO

SET STATISTICS TIME OFF;
SET STATISTICS IO OFF;
GO
Knowledge Check
  1. In a blocking chain, which session should you identify before deciding to KILL?
  2. What does PAGEIOLATCH_SH in wait statistics indicate?
  3. A query plan shows a Table Scan on a million-row table. What should you do?
  4. Which DMV shows all currently executing requests along with their wait type?
  5. In which direction do you read an execution plan diagram in SSMS?
  1. The head blocker — the session with blocking_session_id = 0 in dm_exec_requests that all other blocked sessions trace back to.
  2. SQL Server is waiting to read a page from disk — indicates I/O pressure, usually caused by insufficient memory (buffer pool too small) or slow storage.
  3. Add an index on the columns used in the WHERE clause or JOIN to enable an Index Seek instead of a Table Scan.
  4. sys.dm_exec_requests — join it with sys.dm_exec_sql_text using CROSS APPLY to get the query text.
  5. Right to left — data flows from the leaf operators on the right (data access) to the root operator on the left (final result).
09
Monitor NovaMart and Identify a Blocking Session
~45 min Requires: NovaMart DB · Two SSMS windows Activity Monitor · DMVs
Mission Briefing — Lab 09

Users Report Slow Responses — Investigate NovaMart

End-users are reporting that NovaMart queries are running slowly this afternoon. Your job is to open Activity Monitor, query DMVs to find active requests, simulate a blocking scenario in two SSMS windows, identify the head blocker, and read an execution plan for a slow query.

Part A — Open Activity Monitor

  1. In SSMS, press Ctrl+Alt+A or right-click the server node → Activity Monitor. Expand the Processes and Resource Waits panes.

    Activity Monitor opens and shows active sessions. CPU and Batch Requests/sec charts are visible in the Overview section.
Part B — Query Active Requests via DMV
  1. T-SQL
    SELECT
  r.session_id,
  r.status,
  r.wait_type,
  r.cpu_time,
  r.total_elapsed_time / 1000  AS ElapsedSec,
  r.blocking_session_id,
  SUBSTRING(t.text, 1, 200) AS QueryText
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.session_id > 50
ORDER BY r.total_elapsed_time DESC;
    Result shows your own SELECT statement. blocking_session_id is 0 (no blocking yet).
Part C — Simulate Blocking

  1. Window 1 (Blocker): Open a new query window and run — do NOT commit or close:

    T-SQL — Window 1
    USE NovaMart;
GO
BEGIN TRANSACTION;
UPDATE dbo.Employee SET Salary = Salary + 1 WHERE EmpID = 1;
-- Do NOT commit — leave this window open

  2. Window 2 (Victim): Open a second query window and run:

    T-SQL — Window 2
    USE NovaMart;
GO
SELECT * FROM dbo.Employee WHERE EmpID = 1;
    Window 2 hangs (spinning) — it is blocked waiting for Window 1 to release its lock.

  3. Window 3 (Monitor): Open a third query window and identify the head blocker:

    T-SQL — Window 3
    SELECT
  r.session_id AS BlockedSPID,
  r.blocking_session_id AS BlockedBy,
  r.wait_type,
  SUBSTRING(t.text, 1, 200) AS BlockedQuery
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.blocking_session_id <> 0;
    One row appears showing Window 2's SPID blocked by Window 1's SPID.

  4. Resolve the blocking — go back to Window 1 and commit:

    T-SQL — Window 1
    COMMIT;
GO
    Window 2 immediately returns its result. Window 3 DMV query shows no more blocked sessions.
Part D — View Execution Plan

  1. In a query window, press Ctrl+M to enable Actual Execution Plan, then run:

    T-SQL
    USE NovaMart;
GO
SELECT e.FullName, d.DeptName, e.Salary
FROM   dbo.Employee   e
JOIN   dbo.Department d ON e.DeptID = d.DeptID
WHERE  e.Salary > 5000000;
    Execution Plan tab appears. Click the plan operators to see row counts and cost percentages. Note whether the plan uses Index Seek or Table Scan on the Employee table.

Validation Checklist

  • Activity Monitor opened and Processes pane visible
  • dm_exec_requests query returned active sessions
  • Blocking scenario simulated — Window 2 hung waiting
  • Head blocker identified via dm_exec_requests WHERE blocking_session_id <> 0
  • Blocking resolved by committing the transaction in Window 1
  • Execution plan viewed — operators identified (Seek vs Scan)

Troubleshooting

ProblemLikely CauseFix
Activity Monitor shows "Refresh paused"Monitor was manually paused or server is under very high loadClick Resume in the Activity Monitor toolbar
Window 2 does not hang — returns immediatelyREAD COMMITTED SNAPSHOT ISOLATION (RCSI) is enabled on NovaMartDisable RCSI: ALTER DATABASE NovaMart SET READ_COMMITTED_SNAPSHOT OFF
