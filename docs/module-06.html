Module 06: Backup & Restore

Lab 06 Recovery Models · BACKUP · RESTORE NovaMart Production
Learning Objectives
  • Explain the difference between Simple and Full recovery models
  • Perform a full database backup using SSMS and T-SQL
  • Perform a differential backup to capture incremental changes
  • Back up and restore the transaction log to a point in time
  • Restore NovaMart from a backup file and verify data integrity

6.1 Recovery Models

The recovery model controls how much of the transaction log SQL Server retains — and therefore, what you can restore. Set it deliberately before your first backup.

ModelLog RetentionPoint-in-Time Restore?Use Case
SIMPLETruncated at each checkpoint automaticallyNoDevelopment, test, non-critical databases
FULLKept until a log backup is takenYesProduction — required for NovaMart
BULK-LOGGEDMinimally logs bulk operationsNo (during bulk period)Temporary switch during large data loads
⚠
SIMPLE Recovery = No Log Backups

In SIMPLE recovery mode, SQL Server truncates the log automatically. You cannot take log backups. Maximum potential data loss equals all changes since the last full or differential backup. Never use SIMPLE on a production database that requires point-in-time recovery.

T-SQL — Set Recovery Model
-- Set NovaMart to FULL recovery (required for production)
ALTER DATABASE NovaMart SET RECOVERY FULL;
GO

-- Verify the current recovery model
SELECT name, recovery_model_desc
FROM   sys.databases
WHERE  name = 'NovaMart';

6.2 Backup Types

SQL Server supports three standard backup types. Use a combination to balance storage cost against recovery time and maximum data loss.

TypeWhat It CapturesRestore DependencyTypical Frequency
FullEntire database — data pages plus enough log to make it consistentSelf-contained — can restore from this file aloneWeekly or nightly
DifferentialAll pages changed since the last full backupRequires the last full backup firstNightly or every few hours
Transaction LogActive portion of the transaction logRequires full + any differentials firstEvery 15–60 min in production
ℹ
NovaMart Backup Strategy

Full backup Sunday 02:00 → Differential Mon–Sat 00:00 → Transaction log every 30 minutes. This limits maximum data loss to 30 minutes and keeps daily backup files small.

6.3 Performing Backups via T-SQL

T-SQL backup commands are scriptable, schedulable, and the foundation for SQL Agent jobs. Master these before relying on the GUI.

T-SQL — Full Backup
-- Full backup
BACKUP DATABASE NovaMart
TO DISK = 'C:\SQLBackup\NovaMart_Full.bak'
WITH FORMAT,               -- overwrite and initialise the media
     INIT,                 -- overwrite existing backup sets on the file
     NAME = 'NovaMart Full Backup',
     COMPRESSION,          -- reduces file size — recommended
     STATS = 10;           -- print progress every 10%
GO
T-SQL — Differential Backup
-- Differential backup (captures changes since last FULL)
BACKUP DATABASE NovaMart
TO DISK = 'C:\SQLBackup\NovaMart_Diff.bak'
WITH DIFFERENTIAL,
     INIT,
     NAME = 'NovaMart Differential Backup',
     COMPRESSION,
     STATS = 10;
GO
T-SQL — Transaction Log Backup
-- Transaction log backup (only valid in FULL or BULK-LOGGED recovery)
BACKUP LOG NovaMart
TO DISK = 'C:\SQLBackup\NovaMart_Log.bak'
WITH INIT,
     NAME = 'NovaMart Log Backup',
     STATS = 10;
GO

-- Verify a backup file without restoring
RESTORE VERIFYONLY FROM DISK = 'C:\SQLBackup\NovaMart_Full.bak';
GO
OptionMeaning
FORMAT, INITOverwrite existing backup file. Remove INIT to append to the same file (multiple backup sets).
COMPRESSIONReduces backup file size significantly — recommended unless storage is fast SSD.
STATS = 10Print a progress message every 10% — useful for large databases.
DIFFERENTIALCaptures only pages changed since the last full backup — not since last diff.
BACKUP LOGOnly valid when recovery model is FULL or BULK-LOGGED.

6.4 Restoring a Database

A backup is worthless unless you have practised the restore. Restoring in SQL Server follows a strict sequence: full → differential → log backups, each with NORECOVERY except the last.

⚠
NORECOVERY vs RECOVERY

WITH NORECOVERY leaves the database in a Restoring state so you can apply more backups. WITH RECOVERY brings the database online — use it only on the final step. Applying RECOVERY too early means you cannot apply further log backups.

T-SQL — Full Restore Sequence
-- Step 1: Restore full backup — leave in restoring state
RESTORE DATABASE NovaMart
FROM DISK = 'C:\SQLBackup\NovaMart_Full.bak'
WITH NORECOVERY,
     STATS = 10;
GO

-- Step 2: Apply differential backup
RESTORE DATABASE NovaMart
FROM DISK = 'C:\SQLBackup\NovaMart_Diff.bak'
WITH NORECOVERY,
     STATS = 10;
GO

-- Step 3: Apply log backup — bring database online
RESTORE LOG NovaMart
FROM DISK = 'C:\SQLBackup\NovaMart_Log.bak'
WITH RECOVERY;
GO
T-SQL — Point-in-Time Restore
-- Restore to a specific moment (FULL recovery model required)
RESTORE LOG NovaMart
FROM DISK = 'C:\SQLBackup\NovaMart_Log.bak'
WITH RECOVERY,
     STOPAT = '2026-05-14T09:30:00';
GO

-- Inspect what backup sets are inside a file before restoring
RESTORE HEADERONLY FROM DISK = 'C:\SQLBackup\NovaMart_Full.bak';
GO

-- Restore to a different path (relocate files)
RESTORE DATABASE NovaMart
FROM DISK = 'C:\SQLBackup\NovaMart_Full.bak'
WITH RECOVERY,
     MOVE 'NovaMart'     TO 'D:\SQLData\NovaMart.mdf',
     MOVE 'NovaMart_log' TO 'E:\SQLLog\NovaMart.ldf',
     REPLACE;
GO
OptionMeaning
NORECOVERYLeaves DB in restoring state — required when more backups follow.
RECOVERYBrings database online — only on the FINAL restore step.
STOPATPoint-in-time recovery — FULL recovery model only.
REPLACEOverwrite an existing database with the same name.
MOVERedirect data/log files to different paths during restore.

6.5 Backup Verification

Always verify backups after creation. RESTORE VERIFYONLY checks that the backup set is complete and readable without actually restoring data.

T-SQL — Verify and Inspect Backups
-- Verify backup file integrity
RESTORE VERIFYONLY FROM DISK = 'C:\SQLBackup\NovaMart_Full.bak';
GO

-- View backup history for NovaMart
SELECT bs.database_name, bs.type,
       bs.backup_start_date, bs.backup_finish_date,
       CAST(bs.backup_size / 1048576.0 AS DECIMAL(10,1)) AS SizeMB,
       bmf.physical_device_name
FROM   msdb.dbo.backupset bs
JOIN   msdb.dbo.backupmediafamily bmf
       ON bs.media_set_id = bmf.media_set_id
WHERE  bs.database_name = 'NovaMart'
ORDER  BY bs.backup_start_date DESC;
Knowledge Check
  1. NovaMart is in SIMPLE recovery. Can you take a transaction log backup?
  2. You restored a full backup WITH NORECOVERY and then applied the log backup WITH RECOVERY. Can you apply another log backup now?
  3. What does RESTORE VERIFYONLY confirm?
  4. How often should you run log backups on a production database?
  5. What SQL Server option limits the maximum data loss during an outage when using FULL recovery?
  1. No. Transaction log backups require FULL or BULK-LOGGED recovery model. In SIMPLE, the log is truncated automatically and cannot be backed up.
  2. No. Once you apply WITH RECOVERY, the database comes online and the restore chain is broken. You cannot apply any further log backups.
  3. RESTORE VERIFYONLY confirms the backup set is complete and readable, but does not restore any data. It validates the media, not the actual row data.
  4. Every 15–60 minutes in a production environment, depending on the acceptable RPO (Recovery Point Objective).
  5. The frequency of transaction log backups. The maximum data loss equals the time between the last successful log backup and the failure.

Back Up and Restore the NovaMart Database
~60 min Requires: NovaMart DB · C:\SQLBackup folder BACKUP · RESTORE
Mission Briefing — Lab 06

Protect NovaMart Before the Weekend

It is Friday afternoon. Budi needs to confirm that NovaMart has a working backup-and-restore procedure before end of business. Your tasks:

  • Set the recovery model to FULL
  • Take a full backup, add some data, take a differential
  • Simulate a failure by dropping a table
  • Restore from backup and confirm all data returns
Part A — Prepare

  1. Create the backup directory if it does not exist:

    PowerShell
    New-Item -ItemType Directory -Force -Path 'C:\SQLBackup'

  2. Ensure NovaMart is in FULL recovery:

    T-SQL
    ALTER DATABASE NovaMart SET RECOVERY FULL;
GO
SELECT name, recovery_model_desc FROM sys.databases WHERE name = 'NovaMart';
    recovery_model_desc shows FULL.
Part B — Full Backup
  1. T-SQL
    BACKUP DATABASE NovaMart
TO DISK = 'C:\SQLBackup\NovaMart_Full.bak'
WITH FORMAT, INIT,
     NAME = 'NovaMart Full Backup',
     COMPRESSION, STATS = 10;
GO
    Processed N pages… BACKUP DATABASE successfully processed. File NovaMart_Full.bak appears in C:\SQLBackup.

  2. Verify the backup:

    T-SQL
    RESTORE VERIFYONLY FROM DISK = 'C:\SQLBackup\NovaMart_Full.bak';
    The backup set on file 1 is valid.
Part C — Make a Change, Then Differential Backup

  1. Insert a test row to simulate activity after the full backup:

    T-SQL
    USE NovaMart;
GO
INSERT INTO dbo.Department (DeptName, Location)
VALUES ('Logistics', 'Warehouse B');
GO
SELECT * FROM dbo.Department;
  2. T-SQL
    BACKUP DATABASE NovaMart
TO DISK = 'C:\SQLBackup\NovaMart_Diff.bak'
WITH DIFFERENTIAL, INIT,
     NAME = 'NovaMart Differential Backup',
     COMPRESSION, STATS = 10;
GO
    BACKUP DATABASE WITH DIFFERENTIAL successfully processed.
Part D — Simulate Failure

  1. Drop the Department table to simulate accidental data loss:

    T-SQL
    USE NovaMart;
GO
DROP TABLE dbo.Department;
GO
-- Confirm it is gone
SELECT name FROM sys.tables WHERE name = 'Department';
    No rows returned — table is gone.
Part E — Restore from Backup

  1. Kick all connections out of NovaMart, then restore full backup:

    T-SQL
    USE master;
GO
ALTER DATABASE NovaMart SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
GO
RESTORE DATABASE NovaMart
FROM DISK = 'C:\SQLBackup\NovaMart_Full.bak'
WITH NORECOVERY, REPLACE, STATS = 10;
GO

  2. Apply the differential backup, then bring online:

    T-SQL
    RESTORE DATABASE NovaMart
FROM DISK = 'C:\SQLBackup\NovaMart_Diff.bak'
WITH RECOVERY, STATS = 10;
GO
ALTER DATABASE NovaMart SET MULTI_USER;
GO
    RESTORE DATABASE successfully processed. NovaMart is online.

  3. Confirm the Department table and the Logistics row are back:

    T-SQL
    USE NovaMart;
GO
SELECT * FROM dbo.Department;
    Department table exists and includes the 'Logistics' row added before the differential backup.

Validation Checklist

  • NovaMart recovery model set to FULL
  • Full backup created and verified with RESTORE VERIFYONLY
  • Differential backup created after inserting Logistics row
  • Department table successfully dropped (failure simulated)
  • Full + differential restore completed without error
  • Department table and Logistics row present after restore

Troubleshooting

ProblemLikely CauseFix
BACKUP DATABASE fails — cannot open fileC:\SQLBackup directory does not exist or SQL Server service account lacks write permissionCreate the directory and grant write rights to the SQL Server service account (check in Configuration Manager)
RESTORE fails — database in useActive connections existRun ALTER DATABASE NovaMart SET SINGLE_USER WITH ROLLBACK IMMEDIATE before restoring
Cannot apply differential — "The log or differential backup cannot be restored"Differential is from a different full backup chainRestore the correct full backup first; the differential must be newer than the full backup you restored
BACKUP LOG fails — "not permitted under the simple recovery model"Recovery model is SIMPLERun ALTER DATABASE NovaMart SET RECOVERY FULL
