Storage Planning

Separate data, log, and backup files onto different physical disks to maximize I/O throughput and protect against disk failures.

File TypeExtensionBest Practice
Data files.mdf / .ndfRAID 10 or SSD volume
Log files.ldfSeparate volume — sequential writes benefit from fast sequential I/O
TempDB.mdf / .ldfFastest available disk; pre-size to avoid autogrowth
Backups.bakSeparate volume or network path — never on same disk as data

Collation

Collation determines how SQL Server sorts and compares character data (case sensitivity, accent sensitivity, etc.). Choose carefully before installation — changing collation after the fact is complex.

CollationNotes
SQL_Latin1_General_CP1_CI_ASCommon default in many installations. CI = Case Insensitive, AS = Accent Sensitive
Latin1_General_CI_ASWindows collation — preferred for new installations
Latin1_General_CS_ASCase Sensitive — 'Admin' ≠ 'admin'
⚠
Training VM Collation

The training VM uses the installer default. Do not change collation during the lab. Note the value during installation for documentation purposes.

Service Accounts

Each SQL Server service should run under a dedicated, least-privilege account. Avoid using a local Administrator or Domain Admin account.

ServiceRecommended Account
SQL Server (Database Engine)Managed Service Account (MSA) or NT SERVICE\MSSQLSERVER
SQL Server AgentManaged Service Account or NT SERVICE\SQLSERVERAGENT
SQL Server BrowserNT AUTHORITY\LOCAL SERVICE

2.2 Installation Walkthrough

SQL Server is installed from the Installation Center, launched by running setup.exe from the SQL Server ISO or media folder.

ℹ
Lab Note

SQL Server is already installed on the training VM. Walk through this section with your instructor using the slides. In the lab (Part A), you will verify the installed instance and review Configuration Manager instead of running setup.exe again.

Key Installation Steps

  1. Run setup.exe → Installation Center opens.
  2. Click Installation → New SQL Server stand-alone installation.
  3. Product Key — enter key or choose Developer/Evaluation edition.
  4. License Terms — accept the EULA.
  5. Microsoft Update — optionally check for updates.
  6. Install Rules — setup checks prerequisites. Fix any failures before continuing.
  7. Feature Selection — at minimum, select: Database Engine Services, SQL Server Replication (optional), Management Tools – Basic (if SSMS not already installed).
  8. Instance Configuration — choose Default or Named instance. Record the instance name.
  9. Service Accounts — assign service accounts and set startup type.
  10. Database Engine Configuration:
    • Authentication Mode — choose Windows or Mixed Mode.
    • Add Current User to sysadmin (required).
    • Data Directories — set data, log, backup, and tempdb paths.
    • TempDB tab — set number of data files (guideline: match core count, max 8).
  11. Ready to Install — review configuration summary then click Install.

Installation Duration

A typical lab-grade installation completes in 5–15 minutes. Production installs with all features can take 30–60 minutes.

2.3 Post-Installation Configuration

Verify Service Startup Type

Open SQL Server Configuration Manager (see Section 2.4). Verify that the Database Engine and SQL Server Agent services have Startup Type = Automatic so they restart after a server reboot.

Enable the sa Login

If you chose Mixed Mode, the sa login is disabled by default for security. Enable it only if required:

T-SQL
-- Enable sa and set a strong password
ALTER LOGIN sa ENABLE;
ALTER LOGIN sa WITH PASSWORD = 'Str0ng!P@ssw0rd';
GO
⚠
Security Best Practice

Keep sa disabled unless specifically required. Use named SQL logins or Windows Authentication for all application connections.

Set Maximum Server Memory

SQL Server will, by default, consume nearly all available RAM. Set a maximum to leave memory for the OS and other services:

T-SQL
-- Example: cap SQL Server at 12 GB on a 16 GB server
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'max server memory (MB)', 12288;
RECONFIGURE;
GO

2.4 SQL Server Configuration Manager

Configuration Manager is the authoritative tool for managing SQL Server services and network protocols. Do not use services.msc to start/stop SQL Server — Configuration Manager is aware of clustered instances and dependent services.

ℹ
Opening Configuration Manager

Search for SQL Server Configuration Manager in the Start menu, or run: SQLServerManager16.msc (SQL Server 2022) / SQLServerManager15.msc (SQL Server 2019).

NodeWhat You Can Do
SQL Server ServicesStart/Stop/Restart services; change service accounts; set startup type
SQL Server Network ConfigurationEnable/disable protocols (TCP/IP, Named Pipes, Shared Memory) per instance
SQL Native Client ConfigurationClient-side protocol order and aliases

Enable TCP/IP

  1. Expand SQL Server Network Configuration → Protocols for MSSQLSERVER.
  2. Right-click TCP/IP → Enable.
  3. Double-click TCP/IP → IP Addresses tab → scroll to IPAll.
  4. Set TCP Port = 1433 for a default instance (or leave blank to use the dynamic port).
  5. Click OK.
  6. Restart the SQL Server service for the change to take effect.

Windows Firewall Rule

Even after enabling TCP/IP inside SQL Server, Windows Firewall may block external connections. Create an inbound rule:

PowerShell (run as Administrator)
New-NetFirewallRule -DisplayName "SQL Server 1433" `
    -Direction Inbound -Protocol TCP -LocalPort 1433 `
    -Action Allow

Configure Services, TCP/IP & Verify Remote Connectivity
~45 min sysadmin + local admin Config Manager
Mission Briefing — Lab 02

Opening the Door for Remote Users

Sari (HR) and Dian (Finance) sit at separate workstations across the building. Budi has assigned them Windows accounts and asked you to make sure they can connect to the SQL Server remotely using SSMS.

Before you create any databases, you need to make sure the network layer is ready. Today's tasks: verify services are running, enable TCP/IP, configure the firewall, and test a remote connection.

Part A — Verify Service Status and Startup Type

  1. Open SQL Server Configuration Manager.

  2. Click SQL Server Services. Verify the following:

    ServiceExpected StateExpected Startup Mode
    SQL Server (MSSQLSERVER)RunningAutomatic
    SQL Server Agent (MSSQLSERVER)Running or StoppedAutomatic

  3. If the startup mode for either service is not Automatic, right-click it → Properties → Service tab → Start Mode → Automatic → OK.

Part B — Enable TCP/IP Protocol

  1. In Configuration Manager, expand SQL Server Network Configuration → Protocols for MSSQLSERVER.

  2. Check the TCP/IP status.

    If already Enabled — note this and skip to Part C.

  3. If Disabled — right-click TCP/IP → Enable.

  4. Double-click TCP/IP → IP Addresses tab → scroll to IPAll → set TCP Port = 1433. Click OK.

  5. Right-click SQL Server (MSSQLSERVER) → Restart to apply the protocol change.

    Service shows Running again after restart.
Part C — Create a Windows Firewall Rule

  1. Open Windows Defender Firewall with Advanced Security from the Start menu.

  2. Click Inbound Rules → New Rule.

  3. Follow the wizard: Port → TCP → Specific local ports: 1433 → Allow the connection → all profiles → Name: "SQL Server 1433". Click Finish.

    A new inbound rule named "SQL Server 1433" appears in the list with green check.
Part D — Verify TCP/IP Port via T-SQL

  1. In SSMS, run:

    T-SQL
    -- View current network listener information
SELECT local_tcp_port, local_net_address
FROM   sys.dm_exec_connections
WHERE  session_id = @@SPID;
    Returns the port (1433) and IP address your current SSMS session is using.
Part E — Test Remote Connection

  1. On the same machine, open a new SSMS connection window: File → Connect Object Explorer.

  2. In the Server name field, type the explicit IP address with port: 127.0.0.1,1433. Leave Authentication as Windows Authentication. Click Connect.

    SSMS connects successfully — Object Explorer shows a second entry for the same instance connected via IP:port.

Validation Checklist

  • SQL Server service is Running with Startup Mode = Automatic
  • SQL Server Agent startup mode is set to Automatic
  • TCP/IP is Enabled in Configuration Manager
  • IPAll → TCP Port = 1433
  • Firewall inbound rule "SQL Server 1433" exists
  • SSMS connects via 127.0.0.1,1433 without error
